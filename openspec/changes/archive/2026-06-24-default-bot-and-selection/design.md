## Context

智能客服系统目前所有用户共用一个机器人（通过 `bot_id=c72c47d8...` 硬编码）。系统需要支持多机器人：
- 不同行业/场景使用不同配置的机器人
- Admin 可以设置哪个机器人作为默认
- 客户端用户可以自主选择机器人

**当前状态**:
- `bots` 表结构：`id, name, description, industry_type, status, created_at, updated_at`
- 客户端硬编码使用 `default-bot`（不存在）
- 用户认证使用简单的 token 方式

**约束**:
- 尽量减少数据库变更
- 兼容现有 API 契约
- 前端改动不能影响未登录用户的体验

## Goals / Non-Goals

**Goals:**
- Admin 可以在 bot 列表中设置默认机器人
- 未登录用户自动使用默认机器人
- 登录用户可以自主选择任意机器人
- 提供获取机器人列表的 API
- **不同机器人的知识库和 QA 数据相互隔离**

**Non-Goals:**
- 不实现机器人的访问权限控制（谁可以用哪个机器人）
- 不修改现有的机器人配置 CRUD API

## Decisions

### 1. 数据库变更: 添加 `is_default` 字段

**选择**: 在 `bots` 表添加 `is_default BOOLEAN DEFAULT 0`

**理由**:
- 简单直接，每个 bot 知道自己是否是默认
- 避免创建单独的 `default_bot` 配置表
- ALTER TABLE 对现有数据影响小

**备选考虑**:
- 创建 `bot_settings` 表存储默认机器人 → 增加复杂度，当前不需要
- 使用 Redis 缓存默认机器人 → 引入新依赖，权衡后不需要

### 2. 知识库和 QA 隔离设计

**当前架构**: 系统已使用 ChromaDB 的多 Collection 机制实现租户隔离
- 知识库分块: `kb_chunks_{bot_id}` - 每个 bot 独立的 collection
- QA 嵌入: `qa_embeddings_{bot_id}` - 每个 bot 独立的 collection

**变更**: 无需修改，当前架构已支持 bot 级别的数据隔离

**理由**:
- ChromaDB 通过 collection name 隔离数据
- SQLite 中 `documents` 和 `qa_pairs` 表通过 `bot_id` 字段隔离
- 聊天时根据 `bot_id` 访问对应 collection/记录

### 3. API 设计

**新增端点**:
- `GET /api/admin/bots/default` - 获取当前默认机器人
- `PUT /api/admin/bots/default/{bot_id}` - 设置默认机器人
- `GET /api/bots` - 获取所有可用机器人列表（给客户端下拉用）

**Chat API 变更**:
- `POST /api/chat` 的 `bot_id` 改为可选：
  - 不传 → 使用默认机器人
  - 传值 → 使用指定的机器人（需验证存在）

### 4. 前端机器人选择组件

**设计方案**:
- 创建 `BotSelector.vue` 组件
- 未登录用户：不显示选择器，使用默认机器人
- 登录用户：显示下拉选择器，可切换

**理由**: 保持简单，不引入复杂的状态管理

### 5. 默认机器人设置 UI

**设计方案**:
- 在 Admin Bot 列表页面，每行添加"设为默认"按钮
- 当前默认机器人卡片右上角显示"默认"标签
- 添加 Bot 时，表单底部添加"设为默认"复选框（默认不选中）
- **只能设置一个默认机器人**：设置新默认时会自动取消旧的默认

**添加第一个机器人时的引导流程**:
1. 管理员提交添加 Bot 表单
2. 如果是第一个 Bot（系统中没有任何机器人），则：
   - Bot 创建成功后，弹出确认对话框
   - 对话框内容："您还没有设置默认机器人。客户在聊天窗口时，如果没有设置默认机器人，将会看到报错信息。"
   - 对话框按钮："取消"（关闭弹窗，不设置默认）| "确认"（设为默认）
3. 如果不是第一个 Bot，不弹窗

**理由**: 引导管理员正确配置，避免客户看到错误

## Risks / Trade-offs

| Risk | Mitigation |
|------|-----------|
| 没有设置默认机器人时，聊天接口报错 | API 层检查，如果未设置默认机器人且请求未指定 bot_id，返回 400 错误 |
| 用户切换机器人后 session 状态不一致 | session 绑定 bot_id，切换时创建新 session 或提示用户 |
| 机器人被删除但仍是默认 | Admin 删除时检查，如果是默认机器人则不允许删除或自动切换 |

## Migration Plan

1. **数据库迁移**: 添加 `is_default` 字段
2. **API 实现**: 实现默认机器人获取/设置 API
3. **Admin 页面**: 添加"设为默认"按钮
4. **客户端 API**: 实现 `/api/bots` 列表接口
5. **客户端页面**: 添加机器人选择器
6. **测试**: 端到端测试默认机器人和切换功能

## Open Questions

1. **Q: 删除机器人时如果它是默认的怎么办？**
   A: 不允许删除，或自动将另一个机器人设为默认

2. **Q: 未登录用户的 session 如何处理？**
   A: 未登录用户的 session_id 仍然有效，每次使用默认机器人

3. **Q: 是否需要限制非管理员设置默认机器人？**
   A: 当前方案假设只有 Admin 用户可以访问 `/api/admin/*` 端点
